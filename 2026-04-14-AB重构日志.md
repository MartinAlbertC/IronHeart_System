- 支持参数：`--id`、`--name`、`--faces`（图片目录）、`--voices`（音频目录）、`--wearer`、`--list`
- 使用 InsightFace 提取人脸 embedding，VoiceEmbedder 提取声纹 embedding

#### `a_layer/src/vision/re_id.py` — Re-ID 模块
- `ReIDModule`：维护 `track_id → alias` 稳定映射
- 三层策略：① 已有映射直接返回 → ② 与 `lost_tracks` 缓冲匹配（TTL 5分钟）→ ③ 查注册库或分配新 `stranger_xxx`
- EMA（指数移动平均，α=0.3）平滑更新 alias embedding
- 高置信度匹配（≥0.7）自动回写注册库
- 新陌生人自动调用 `registry.register_person()` 持久化

#### `a_layer/src/vision/asd.py` — Active Speaker Detection
- 封装 Light-ASD 模型（权重：`models/Light-ASD/weight/pretrain_AVA_CVPR.model`）
- 输入：`{track_id: (T, 112, 112) 嘴部帧序列}` + 16kHz 单声道音频
- 输出：`{track_id: speaking_probability}`
- `extract_mouth_roi()`：从 BGR 帧裁剪嘴部区域

#### `b_layer/identity_fusion.py` — 身份融合模块
- `IdentityFusion.fuse(face_alias, face_conf, voice_alias, voice_conf)`
- 融合逻辑：一致 → 直接使用；单模态 → 直接使用；冲突 → 取置信度更高的
- `fuse_event(event)`：对 A 层事件直接应用融合，返回 `(resolved_alias, modality_used)`

## 二、C 层与 A/B 层协同建议

### 背景

重构后 A/B 层已在事件中携带稳定的 `alias` 字段（如 `zhangsan`、`stranger_a1b2c3d4`），该 alias 由 Re-ID + 双模态注册库保证跨帧稳定性。

当前 C 层的 `IdentityStore.match_or_create()` 在收到事件后**再次**用 embedding 做相似度匹配，与 A/B 层形成双重匹配，存在以下问题：

| 问题 | 说明 |
|------|------|
| 冗余匹配 | 同一人被匹配两次，浪费计算 |
| 阈值不一致 | A/B 层 face 阈值 0.5，C 层 0.75，可能对同一人产生不同判断 |
| ID 不同步 | A/B 层用 `person_id`（语义名），C 层用 `entity_0001`（自增），两套 ID 无法对应 |

---

### 建议改动

#### 建议 1：优先使用 A/B 层传来的 alias（核心建议）

B 层事件的 `payload` 中已包含 `alias` 字段。C 层应优先将其作为 `resolved_entity_id`，仅在 `alias` 为 `None` 时才走 `match_or_create()`。

```python
# c_layer 处理 B 层事件时的建议逻辑
alias = event.payload.get("alias")
if alias:
    # 直接用 alias 作为 resolved_entity_id，只做 upsert 更新 embedding
    identity_store.upsert_entity(alias, face_embedding, voice_embedding)
    resolved_id = alias
else:
    # alias 为 None（说话人未识别），走原有 match_or_create
    resolved_id, is_new = identity_store.match_or_create(face_embedding, voice_embedding)
```

需要在 `IdentityStore` 中新增 `upsert_entity(entity_id, face_emb, voice_emb)` 方法，逻辑与现有 `_update_entity` 相同，但以传入的 `entity_id` 为准，不做匹配。

---

#### 建议 2：`stranger_xxx` 的处理策略

A/B 层对未知人员分配 `stranger_xxxxxxxx` 格式的 alias，TTL 7天后自动清理。

C 层建议：
- **接受** `stranger_xxx` 作为合法的 `resolved_entity_id` 存入 PostgreSQL
- 当 A/B 层后续将该 stranger 识别为已知人（alias 变更）时，C 层应支持 `rename_entity_everywhere(old_id=stranger_xxx, new_id=real_name)` 调用（该方法已存在）
- 可在夜间反思流程中批量处理 stranger → real_name 的映射

---

#### 建议 3：统一 embedding 维度

当前 C 层表结构注释写的是 1024 维，但实际 DDL 是：
- `face_embedding vector(512)`（InsightFace ArcFace R100 输出 512 维）
- `voice_embedding vector(256)`（WeSpeaker ResNet34 输出 256 维）

A/B 层输出的 embedding 维度与此一致（512 + 256），**无需修改**，但建议删除代码注释中错误的"1024维"描述，避免混淆。

---

#### 建议 4：降级模式下的 alias 透传

当前降级模式（PostgreSQL 不可用）下，C 层用自增计数器生成 `entity_0001`，与 A/B 层的 alias 完全脱节。

建议降级模式也优先使用传入的 alias：

```python
# 降级模式建议
if not self._db_available:
    alias = event.payload.get("alias")
    return alias if alias else f"entity_{self._degraded_counter:04d}", True
```

### 改动优先级

| 优先级 | 建议 | 工作量 |
|--------|------|--------|
| 高 | 建议1：alias 优先透传 + 新增 `upsert_entity` | 小（~30行） |
| 中 | 建议4：降级模式 alias 透传 | 极小（~5行） |
| 低 | 建议2：stranger 生命周期管理 | 中（需与夜间反思流程联调） |
| 低 | 建议3：修正注释中的维度描述 | 极小 |
