### 2.2 WORKING 状态

| 项目 | 结果 |
|------|------|
| 条件 | 队列中有积压消息 (opportunities: 2) |
| 状态 | **通过** |
| 返回 | `state: "WORKING", states: ["WORKING"]` |

| 验证 | `status.py` 第 46-47 行: `if any(v > 0 for v in queues.values()): is_working = True` → 队列消息数 > 0 时正确切换为 WORKING |

### 2.3 ORGANIZING 状态

| 项目 | 结果 |
|------|------|
| 条件 | `_reflecting = True` |
| 状态 | **通过** (代码逻辑验证) |
| 返回 | `state: "ORGANIZING", states: ["ORGANIZING"]` |
| 测试方法 | 直接在 API 进程内设置 `_reflecting = True` 后调用 `get_system_state()` |
| 验证 | `status.py` 第 54-55 行: `if _reflecting: states.append("ORGANIZING")` → 正确切换 |
| 备注 | 通过 HTTP 外部轮询无法捕获该状态（见下方说明） |

**ORGANIZING 状态的 HTTP 捕获限制**:

`POST /api/reflect/trigger` 是同步阻塞接口——反思在请求线程内执行，响应返回时反思已完成。因此外部 HTTP 轮询无法在反思执行期间捕获到 ORGANIZING 状态。

内部验证方法：
1. 直接调用 `run_reflection()` 并在子线程中检查 `_reflecting` 标志 → 确认在反思期间 `_reflecting = True`
2. 手动设置 `_reflecting = True` 后调用 `get_system_state()` → 返回 `state: "ORGANIZING"`

**生产环境中**：自动反思由 `ReflectionScheduler` 在后台线程触发，执行期间其他 HTTP 请求可以正常访问 API，此时 `/api/status` 会正确返回 ORGANIZING 状态。当前测试环境下反思执行过快（< 0.5s），导致难以通过 HTTP 轮询捕获。

## 已知问题

### 1. rebuild_db 未完全清理旧数据

**影响**: video_jobs、action_cards、reflection_history 等使用 `CREATE TABLE IF NOT EXISTS` 的表在 rebuild 后保留了旧数据。

**建议**: 在 `rebuild_db.py` 中，对这些表也添加 `DROP TABLE IF EXISTS ... CASCADE`，然后重建。

### 2. GET /api/entity/list 返回 500

**接口**: `GET /api/entity/list`
**返回**: `{"detail":"Entity list not found"}`
**原因**: `IdentityStore.list_all_entities()` 在无数据或 identity 相关表不完整时抛出异常。
**建议**: 检查 `IdentityStore` 实现，确保空结果时返回空列表而非异常。

### 3. GET /api/entity/{entity_id} 返回 500

**接口**: `GET /api/entity/entity_0001`
**原因**: 与上一条相同，IdentityStore 依赖的数据不完整。
**建议**: 同上。

### 4. MIN_INTERVAL_MINUTES 未在自动调度中实际执行

**状态**: 常量已定义 (`DEFAULT_MIN_INTERVAL_MINUTES = 60`)，API 返回中展示，但 `ReflectionScheduler._loop()` 中未实际检查两次自动反思之间的最小间隔。
**建议**: 在 `_loop()` 中添加 `last_auto_reflect_time` 记录，自动触发前检查是否满足最小间隔。
