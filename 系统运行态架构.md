# IronHeart 系统运行态架构

## 一、总体运行形态

系统启动后，各层和中间件以 **独立 OS 进程** 的形式运行，通过 **TCP 端口** 进行通信。`start_all.py` 通过 `subprocess.Popen` 逐个拉起所有进程。

```
┌─────────────────────────────────────────────────────────────────┐
│  操作系统                                                        │
│                                                                 │
│  PID 1001  ┌─────────────────────┐                              │
│            │  python broker.py    │  ← 独立进程，监听 TCP :6380   │
│            │  (asyncio TCP Server)│                              │
│            └────────┬────────────┘                              │
│                     │ TCP :6380                                  │
│           ┌─────────┼─────────┬──────────┬──────────┐           │
│           │         │         │          │          │           │
│  PID 1002 ▼  PID 1003 ▼  PID 1004 ▼  PID 1005 ▼  PID 1006    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────┐ │
│  │ python   │ │ python   │ │ python   │ │ python   │ │ node  │ │
│  │A层/run.py│ │B层/run.py│ │C层/run.py│ │D层/run.py│ │E层    │ │
│  │          │ │          │ │          │ │          │ │       │ │

│  │ vision   │ │ MQ-Sub   │ │ C-API    │ │ 主循环   │ │主循环 │ │

│                            │ (daemon) │                        │
```

## 二、各进程详解

### 2.1 MQ Broker（消息中间件）

| 项目 | 说明 |
|------|------|
| **启动命令** | `python message_queue/broker.py` |
| **进程类型** | Python asyncio 进程 |
| **监听端口** | `0.0.0.0:6380`（TCP Server） |
| **内部模型** | 单线程 asyncio 事件循环 |
| **通信协议** | 自定义 TCP 协议（Newline-delimited JSON） |
| **角色** | 所有层间数据的唯一中转站 |

Broker 本身不做任何业务处理，只负责消息的排队和转发。它在内存中维护 4 个 FIFO 队列：

```
Broker 内存结构:
  queues = {
    "a_events":        [msg1, msg2, ...],   ← A 层写入，B 层消费
    "b_events":        [msg1, msg2, ...],   ← B 层写入，C 层消费
    "opportunities":   [msg1, msg2, ...],   ← C 层写入，D 层消费
    "execution_plans": [msg1, msg2, ...],   ← D 层写入，E 层消费
  }
```

**零拷贝投递**：当消费者已经在等待时，消息不进队列，直接通过 asyncio.Future 投递。

## 三、MQ 通信模式详解

### 3.1 短连接模型

所有层与 Broker 的通信都采用**短连接**：每次操作新建一个 TCP socket，发一条 JSON，收一条 JSON 响应，立即关闭。

```
publish("a_events", event_dict):
  1. socket.connect(localhost:6380)     ← TCP 三次握手
  2. socket.send({"op":"send","queue":"a_events","data":"{...}"}\n)
  3. socket.recv() → {"status":"ok"}\n
  4. socket.close()                     ← TCP 四次挥手

receive("opportunities"):
  1. socket.connect(localhost:6380)     ← TCP 三次握手
  2. socket.send({"op":"recv","queue":"opportunities"}\n)
  3. socket.recv() → 阻塞等待...       ← Broker 持有连接直到有消息
  4. 收到 {"status":"ok","data":"{...}"}\n
  5. socket.close()                     ← TCP 四次挥手
```

### 3.2 为什么用短连接

`recv` 操作在 Broker 端会**阻塞**（队列为空时通过 asyncio.Future 等待）。如果复用同一连接，发送 `send` 操作就无法进行。短连接让每次操作完全独立，代价只是多一次 TCP 握手。

## 四、端口与网络连接汇总

### 4.1 监听端口（服务器角色）

| 端口 | 进程 | 协议 | 谁会连接它 |
|------|------|------|-----------|
| **6380** | Broker | 自定义 TCP (newline-JSON) | A/B/C/D/E 层 |
| **8000** | C 层 API | HTTP REST | 外部客户端（浏览器、curl） |
| **5432** | PostgreSQL | PostgreSQL 协议 | C 层 |

### 4.2 出站连接（客户端角色）

| 进程 | 连接目标 | 用途 |
|------|----------|------|
| A 层 | → Broker :6380 | publish a_events |
| B 层 | → Broker :6380 | recv a_events / publish b_events |
| C 层 | → Broker :6380 | recv b_events / publish opportunities |
| C 层 | → PostgreSQL :5432 | 身份存储、Tier1/Tier2 查询 |
| D 层 | → Broker :6380 | recv opportunities / publish execution_plans |
| D 层 | → 豆包 API (HTTPS) | 重要性打分、EB 归类 |
| E 层 | → Broker :6380 | recv execution_plans |
| E 层 | → DeepSeek API (HTTPS) | 行动规划、文案生成 |

## 五、启动时序

`start_all.py` 严格按照以下顺序拉起各进程：

```
时间 ──────────────────────────────────────────────────▶

  0s    启动 Broker (subprocess.Popen)
  │       └── 轮询等待 TCP :6380 就绪（最多 30 秒）
  1s    ✓ Broker 就绪
  │
  2s    启动 B 层 (subprocess.Popen)  ──→ 开始订阅 a_events（阻塞等待）
  3s    启动 C 层 (subprocess.Popen)  ──→ 开始订阅 b_events（阻塞等待）+ HTTP API 就绪
  3s    启动 D 层 (subprocess.Popen)  ──→ 开始接收 opportunities（阻塞等待）
  3s    启动 E 层 (subprocess.Popen)  ──→ 开始接收 execution_plans（阻塞等待）
  │
  5s    启动 A 层 (subprocess.Popen)  ──→ 加载模型，开始处理视频
  │       └── 视觉/音频管道生成事件 → publish a_events
  │
  ...    事件开始流动：A → B → C → D → E
  │
  Xs    A 层视频处理完毕，进程退出（returncode=0）
  │     B/C/D/E 层继续运行（等待新事件或 Ctrl+C）
```

**关键设计**：A 层最后启动，确保下游 B/C/D/E 层都已就绪并开始订阅队列后，才开始推送事件。A 层处理完视频后正常退出，不影响其他层。
